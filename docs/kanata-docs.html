KANATA API Interface Specification Frontend ↔ Backend インターフェース仕様

共通仕様

ベースURL

http://localhost:8000   // 開発
<動的ポート>             // プロダクション（Electron sidecar）

環境変数 VITE_API_URL で上書き可能。プロダクションでは window.kanata.getBackendUrl() IPC 経由で取得。

レスポンスエンベロープ

Watchlists / Search エンドポイントは共通エンベロープを返す。Quotes / Fundamentals はエンベロープなしで直接配列を返す（例外）。

typescriptinterface ApiResponse<T> {
  success: boolean;
  data: T | null;       // 失敗時 null
  error: string | null;  // 成功時 null
}

1. ヘルスチェック

GET /api/health

レスポンス 200

json{ "status": "ok" }

2. ウォッチリスト

2-1. 全取得

GET /api/watchlists

レスポンス 200

json{ "success": true, "data": [Watchlist, ...], "error": null }

2-2. 作成

POST /api/watchlists 
typescript (body){ name: string }   // 1〜128文字

レスポンス 201

HTTPdetail原因
409"watchlist name already exists"同名リストが既に存在

2-3. 更新（名前変更 / デフォルト設定）

PATCH /api/watchlists/{list_id} 
typescript (body){
  name?: string          // 1〜128文字
  is_default?: boolean
}

レスポンス 200 — 最新 Watchlist

HTTPdetail原因
404"watchlist not found"指定IDが存在しない
409"watchlist name already exists"同名リストが既に存在

2-4. 削除

DELETE /api/watchlists/{list_id}

レスポンス 200 — { "id": 1 }

HTTPdetail原因
400"cannot delete the last watchlist"最後の1件は削除不可
404"watchlist not found"指定IDが存在しない

2-5. 順序変更

PUT /api/watchlists/reorder 
typescript (body){ ids: number[] }   // 全件必須・新順序

レスポンス 200 — [Watchlist, ...]

HTTPdetail原因
400"ids must match existing watchlists exactly"IDセットが既存と不一致

3. ウォッチリストアイテム

3-1. アイテム追加

POST /api/watchlists/{list_id}/items 
typescript (body){
  symbol: string          // 1〜32文字（大文字化される）
  market?: string         // "US" | "JP"、デフォルト "US"（4桁数字は強制 "JP"）
  display_name?: string   // 最大128文字、省略可
}

レスポンス 201 — items を含む最新 Watchlist

HTTPdetail原因
404"watchlist not found"指定リストが存在しない
409"symbol already in watchlist"同シンボルが既に存在

3-2. アイテム削除

DELETE /api/watchlists/{list_id}/items/{symbol}

レスポンス 200 — 最新 Watchlist

HTTPdetail原因
404"item not found"指定シンボルが存在しない

3-3. アイテム順序変更

PUT /api/watchlists/{list_id}/items/reorder 
typescript (body){ symbols: string[] }   // 全件必須・新順序

レスポンス 200 — 最新 Watchlist

HTTPdetail原因
400"symbols must match existing items exactly"シンボルセットが既存と不一致

4. 相場データ（OHLCV）

GET /api/quotes/{symbol}?timeframe={tf}

symbolpathstringティッカー（例: AAPL, 7203）
timeframequerystring5m 15m 60m 1D 1W 1M（デフォルト: 1D）

6. シンボル検索

GET /api/search?q={query}

クエリパラメータ q: 省略または空文字でプリセット全15件を返す。

typescriptinterface SearchResult {
  code: string;
  name: string;
  market: 'JP' | 'US';
}

検索ロジック

  1. 空クエリ → プリセット全15件を返す
  2. プリセット内でコード/名前前方一致 → 該当プリセットのみ返す
  3. プリセット未マッチ且つクエリ長 ≥ 2 → yfinance.Search で外部検索

フィールド制約

WatchlistItem.market1〜16文字、デフォルト "US"、4桁数字は強制 "JP"
WatchlistItem.display_name最大128文字、省略可
(user_id, watchlist.name)ユニーク制約
(watchlist_id, item.symbol)ユニーク制約

キャッシング

全エンドポイントはプロセス内メモリの TTLCache を使用。Redis 等は不使用。

エラーハンドリング（フロントエンド）

watchlistApi.ts の unwrap() は以下の順でエラーを処理する：

  1. res.ok === false → res.json().detail または statusText をメッセージに Error を throw
  2. body.success === false → body.error をメッセージに Error を throw
  3. body.data === null → "API returned unsuccessful response" を throw

Quotes / Fundamentals / Search エンドポイントのエラーは呼び出し元で個別にハンドリング。
